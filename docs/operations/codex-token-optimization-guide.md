# Carbonet Codex Token Optimization Guide

이 문서는 Codex Launcher (carbonet-codex)를 사용할 때 토큰 비용을 최소화하면서도 정확한 결과를 얻기 위한 모드 선택 가이드를 제공한다.

## 작업 분류 및 권장 모드

토큰을 아끼는 핵심은 **"꼭 필요한 만큼의 컨텍스트만 AI에게 전달하는 것"**이다.

| 작업 분류 | 권장 모드 | 컨텍스트 양 | 주요 용도 |
| :--- | :--- | :--- | :--- |
| **코드 분석 / 질문** | `explain` | 최소 (프롬프트만) | 단일 엔진/메서드 코드 설명, 간단한 문법 질문 |
| **요구사항 탐색** | `ask` | 낮음 | "이 기능이 어디에 구현되어 있지?", "관련 Table 명칭은?" |
| **작업 계획 수립** | `plan` | 중간 (Plan 포함) | "로그인 페이지 리팩토링 계획 세워줘", "수정 범위 파악" |
| **일반 구현 / 로직** | `prompt` | 중간 (부분 요약) | 새로운 서비스 메서드 작성, 간단한 버그 수정 |
| **대규모 수정 / 리팩토링** | `apply` | 높음 (전체 요약) | 여러 파일을 동시에 수정하거나, 복잡한 설계가 필요한 경우 |

## 상세 설명

### 1. `explain` 모드
- **특징**: 현재 세션의 `notes`나 `plan`을 일체 주입하지 않는다.
- **언제 사용하나?**: 이미 보고 있는 코드 한 줄이 궁금할 때, 혹은 프로젝트 상관없는 일반적인 프로그래밍 질문을 할 때.
- **토큰 절약**: 세션 요약이 없으므로 입력 토큰이 가장 적게 발생한다.

### 2. `ask` / `prompt` / `plan` 모드
- **특징**: 현재 세션의 마지막 Job 결과나 하이라이트된 요약(Summary) 위주로 주입한다.
- **언제 사용하나?**: 진행 중인 작업의 흐름을 AI가 알고 있어야 하지만, 전체 코드를 다 읽을 필요는 없을 때.
- **토큰 절약**: 불필요한 파일 내용을 배제하고 최소한의 '진행 상황'만 전달한다.

### 3. `apply` 모드
- **특징**: 세션의 모든 주요 Context와 의도를 AI에게 최대로 전달한다.
- **언제 사용하나?**: "일단 해줘" 식의 자동 실행이나, 상호 의존성이 높은 복잡한 코드를 짤 때.
- **주의**: 가장 많은 토큰을 소모하므로, 수정 범위를 미리 `plan`에서 좁혀놓고 사용하는 것이 좋다.

## 토큰 절약 꿀팁

1. **Session Partitioning**: 한 세션에 너무 많은 작업을 몰아넣지 마라. `resonance-10-session-assignment`에 따라 세션을 분리하면 Summarize 토큰을 아낄 수 있다.
2. **Focusing**: `apply` 모드를 쓰기 전에 질문하고자 하는 파일을 미리 열어두거나(Active Document), 타겟 파일을 프롬프트에 명시하면 AI가 불필요한 파일을 읽지 않는다.
3. **Low-Token Models**: 아주 간단한 코드 정리나 주석 달기는 `codex` 대신 `minimax` (carbonet-minimax-codex) 라벨을 선택해 실행할 수 있다.
